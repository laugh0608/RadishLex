# RadishLex 协作约定

本文件为 RadishLex 仓库中的 AI 协作者与人工协作者提供统一协作规范。
它只约束本仓库的工作方式，不代表其他项目，也不复用其他项目的实现边界。

## 称呼

- 对话开始或结束总结时，请称呼我为 `萝卜SAMA`。

## 语言规范

- 默认使用中文进行讨论、说明、提交总结和开发记录。
- 代码、命令、路径、配置键、类型名、接口名、URL 和外部项目名保留原文。
- 新增文档正文默认使用中文；文件名、目录名和稳定锚点优先使用英文。
- 文档应直接说明结论、边界、下一步和阻塞项，避免堆叠低价值背景。

## 项目定位

- RadishLex 是一个以 Rust 为输入核心、Go 为自部署同步后端、Flutter 为管理界面的中文输入系统。
- 项目目标是本地优先、隐私可信、可跨设备同步、可解释和可删除的个人化中文输入体验。
- v1 不从零重写完整中文输入引擎，优先接入成熟底层引擎，并在 Rust core 中建立稳定抽象、用户词库、候选重排和学习能力。
- 平台端只做系统输入法薄壳，不承载用户词库、同步、排序和隐私策略真相源。

## 文档真相源

`docs/` 与根 `README.md` 是本仓库正式文档源。开始任务时按需读取：

1. `README.md`：项目定位、设计原则、MVP 边界和非目标。
2. `docs/technical-plan.md`：总体架构、平台策略、学习设计、同步方案和风险。
3. `docs/roadmap.md`：阶段路线、交付物和退出标准。
4. `docs/repository-layout.md`：仓库结构、crate / server / app / platform 职责。
5. `docs/privacy-sync.md`：数据分级、加密对象、设备授权、删除语义和威胁模型。

规则：

- 若 README、docs 和实现冲突，先判断是实现偏离文档，还是文档已过期，再统一修正。
- 优先更新已有文档，不为一次性讨论创建大量散文档。
- 架构、协议、阶段目标、隐私边界、平台落地策略或协作规则变化时，必须同步更新对应文档。
- 关键入口文档保持简约，只描述当前事实、稳定入口、阶段边界和必要约束。
- 许可条款以仓库根 `LICENSE` 文件为准，当前采用 RadishLex Source-Available License。

## Agent 协同文件

- `AGENTS.md` 与 `CLAUDE.md` 应保持基本同步。
- 若某个协作文件更新了通用协作规则、执行边界、稳定入口引用或验证约定，另一个文件也应同步更新。
- 两个文件只允许保留与入口名称直接相关的表述差异，不应分叉实际协作规范。

## 外部参考

- 兄弟 Radish 项目只作为协作方式、仓库治理和产品经验参考。
- 默认不跨工作区修改其他项目。
- 不把其他项目的源码、目录结构或实现细节整包迁入本仓库。
- 调研外部输入法、引擎、词库或平台实现时，优先记录行为规格和边界，不复制不兼容许可证源码或测试数据。

## 协作流程

- 开始任务前，先检查工作区状态，并阅读与任务直接相关的文档。
- 若用户明确要求直接修改，且范围清晰、风险可控，则直接实施。
- 若用户没有明确要求直接修改，编写代码前应先说明方案。
- 若需求不明确，或改动会影响架构、协议、隐私边界、阶段范围、平台接入方式、许可证或验证基线，则先说明判断并做必要澄清。
- 每次完成可分割子步骤后，执行匹配的必要验证。
- 重要阶段性决策除了改代码，还应同步更新对应文档；如果属于阶段推进，应记录到开发日志或状态文档。

## AI 执行边界

### 可直接执行

- 读取本仓库代码、文档和配置。
- 修改本仓库内代码、文档、配置和脚本。
- `git status`、`git diff`、`git log` 等只读 Git 操作。
- 项目已存在验证入口时，执行对应格式化、构建、测试和仓库检查命令。
- 明确的小规模提交操作。

### 需要先告知用户再执行

- 安装依赖、下载 SDK / 模型 / 数据、修改全局工具链或需要网络的命令。
- 启动长期运行进程，例如开发服务器、桌面 UI、输入法守护进程或平台服务。
- 修改系统输入法配置、注册 COM / TSF、安装输入法、写系统目录、写注册表、安装证书或修改 Keychain。
- 创建远程仓库、推送远端分支、发布包、打包签名或上传产物。
- 更改许可证、商标、版权声明或第三方依赖授权口径。

### 默认不做

- 跨工作区编辑兄弟 Radish 项目。
- 复制外部项目源码、GPL/LGPL 实现细节或有版权风险的词库数据。
- 未经明确要求执行破坏性 Git 操作。
- 上传原始输入流、明文用户词库、明文候选偏好或敏感上下文。
- 在 v1 阶段把完整拼音引擎自研、全平台输入法端和复杂云服务同时压入主线。

## 架构边界

- Rust core 负责输入会话、候选模型、候选重排、用户词库、个人化学习、同步客户端、加密和 FFI 边界。
- Engine adapter 负责屏蔽底层输入引擎差异；v1 可接 `librime`，但 Rust core 不应依赖具体引擎实现细节。
- Go server 只负责设备管理、密文 blob 同步、版本历史、备份恢复和包分发，不参与按键热路径，不保存明文输入数据。
- Flutter manager 只负责设置、词库管理、学习记录可视化、隐私控制台、同步状态和设备管理，不进入输入热路径。
- 平台壳只负责系统输入法生命周期、按键事件接收、候选窗展示、文本提交和调用 Rust core。
- 候选窗优先使用平台原生机制，不强行统一桌面和移动端候选窗 UI。

## 隐私与同步约束

- 输入热路径必须本地可用，后端不得成为实时输入依赖。
- 服务端默认不可信，只能看到必要元数据和加密对象。
- P0 数据永不同步，包括密码框输入、系统 secure text entry、敏感 App 和用户开启隐私模式期间的输入。
- 原始选择事件、负反馈细节和应用上下文默认只本地学习，除非文档和用户设置明确允许。
- 用户词库、权重摘要、输入方案配置和设备设置只能以加密对象同步。
- 删除必须有可同步语义，避免旧设备或旧备份复活用户已删除词条。
- 日志、测试 fixture 和错误报告不得包含明文输入历史、联系人、密码、证件、支付或其他敏感内容。

## 实现与代码风格

- 新增 Rust、Go、Dart、Swift、Kotlin、C/C++ 或脚本代码时，遵循对应语言主流、清晰、可维护实践。
- 命名必须表达真实领域职责，避免只有技术包装意义的泛化 `manager`、`helper`、`processor`、`context` 或空转 adapter。
- 抽象只在能减少真实重复、隔离稳定边界或表达明确领域概念时引入。
- Rust 代码优先用类型系统、`Result` / `Option`、所有权和小而明确的模块边界表达约束。
- Go server 优先保持简单、可审计、可自部署，默认单用户 SQLite 模式先行。
- Flutter UI 不承载业务真相源，状态与配置边界应通过明确 bridge / API 表达。
- 单个源码文件原则上不超过 1500 行；接近阈值时优先按职责拆分。
- `src/` 下源码按职责做浅层目录分组，避免长期平铺，也避免过深目录树。

## 验证要求

- 仓库级验证入口补齐后，优先使用 `./scripts/check-repo.sh` 或对应平台脚本。
- Rust 改动优先执行 `cargo fmt --check`、`cargo check` 和相关 `cargo test`。
- Go server 改动优先执行 `gofmt`、`go test ./...` 和相关集成检查。
- Flutter manager 改动优先执行 `dart format`、`flutter analyze` 和相关 widget / unit test。
- 隐私、同步、加密、删除语义、候选排序和 FFI 边界改动需要更高强度测试。
- 只改文档或治理文件时，至少确认工作区状态和文件格式没有引入明显问题。

## Git 提交规范

- 使用简洁明确的 Conventional Commits 风格。
- 常用类型：`feat`、`fix`、`docs`、`refactor`、`test`、`chore`、`ci`、`build`、`perf`、`revert`。
- 优先把代码改动、文档改动和仓库治理改动按主题拆分。
- 不添加 AI 协作者署名。
- 提交前至少确认本次改动对应的必要验证已经执行。

## 判断标准

一个改动方向通常应同时满足：

- 本地优先和隐私边界更清晰。
- Rust core、Go server、Flutter manager、平台壳职责没有互相污染。
- 文档、代码和正式口径一致。
- 没有把尚未进入当前阶段的复杂度提前压进主线。
