# 2026-W26 开发日志

本文档记录 2026-W26 周内 RadishLex 的阶段性推进事实、验证证据和后续建议，读者是项目维护者和后续 AI / 人工协作者。本文不承载长期架构设计、接口字段完整参考或历史推演；长期边界应回写到对应专题文档。

## 本周目标

- 从 Phase 0 方案冻结推进到 Phase 1 Rust Core 原型起步。
- 先固化 Rust core 与底层输入引擎之间的稳定边界。
- 建立可复验的 `ime-core` 最小 workspace 基线。

## 完成情况

- 新增 `docs/engine-boundary.md`，明确 engine trait、核心模型、adapter 职责、错误语义、生命周期和 clean-room 约束。
- 新增 Rust workspace 入口 `Cargo.toml` 与 `crates/ime-core/`。
- 在 `ime-core` 中落地 `InputSession`、`Engine`、`KeyEvent`、`Composition`、`Candidate`、`Commit`、`SchemaId` 和 `SessionState`。
- 新增 test-only stub engine，验证 `push_key -> candidates -> commit_candidate` 生命周期。
- 新增 `radishlex-ime-cli demo <input-code> [candidate-index]`，基于合成 demo adapter 复验 core 生命周期。
- 更新 README 与仓库结构文档入口。
- 将 Rust fmt/test 纳入 `scripts/check-repo.py` 仓库级基线。

## 关键决策

- Phase 1 起步只定义平台无关核心模型，不接 `librime`、SQLite、Go server、Flutter 或平台 SDK。
- test-only stub engine 只用于生命周期验证，不作为真实中文输入引擎。
- CLI demo 使用合成候选，只验证 core 调用路径，不代表真实输入质量。
- `ime-core` 不暴露底层 engine 私有对象 ID；后续 ranker 与学习只依赖 RadishLex candidate。
