# 2026-W27 开发日志

本文档记录 2026-W27 周内 RadishLex 的阶段性推进事实、验证证据和后续建议，读者是项目维护者和后续 AI / 人工协作者。本文不承载长期架构设计、接口字段完整参考或历史推演；长期边界应回写到对应专题文档。

## 本周目标

- 继续推进 Phase 3 自部署同步服务端前置治理。
- 在真实 HTTP handler 和远端上传下载前，补齐 Go server storage 层的签名验签门禁。
- 保持服务端默认不可信，只验证 metadata、设备状态、签名、版本冲突和密文完整性。

## 2026-06-29 推进记录

- 已在 `server/sync-server/internal/storage` 新增 Ed25519 签名验证抽象，使用与 Rust `radishlex-signature-v1` 对齐的 length-prefixed canonical field list。
- Go storage 写入前已验证 signed object manifest、device authorization、device revocation 和 recovery record，公钥来源为 `devices.signing_public_key`，并校验 `signature_schema_version`、`signature_algorithm`、`signature_key_id`、签名设备状态和签名时间窗口。
- SQLite metadata schema 已补签名 schema、算法、key id、authorization `join_short_code` 和 recovery KDF 参数字段，用于覆盖签名 canonical bytes 所需 metadata。
- Storage conformance tests 已改为使用确定性 Ed25519 fixture，内存 store 与 SQLite store 共用同一套真实签名 happy path 和字段篡改失败测试。
- 新增篡改拒绝测试覆盖 object manifest、device authorization、device revocation 和 recovery record；签名失败返回 `invalid_signature`，不会提交 metadata 或 blob。
- 已同步 `docs/sync-server-api-storage.md`、`docs/technical-plan.md`、`docs/roadmap.md` 和 `docs/repository-layout.md`，将下一步从签名验证调整为 wrapped device key bytes 承载、recovery wrapped material 读取接口、metadata API 和错误语义验证。
- 已在 storage interface 中将设备授权写入收敛为 `DeviceAuthorizationUpload`，让 authorization metadata、wrapping metadata 和 encrypted wrapped key bytes 作为一个事务输入提交。
- 已为内存 store 和 SQLite store 补齐 wrapped key bytes blob 承载、`DeviceWrappedKey` 读取接口、长度 / `ciphertext_hash` 复验，以及 SQLite 缺失 wrapped key blob 时的 `storage_unavailable` 失败路径。
- 已扩展 blob ref 路径安全测试，覆盖 wrapping blob ref 不泄漏原始 ID / hash 分隔符。
- 已同步 `docs/sync-server-api-storage.md`、`docs/technical-plan.md`、`docs/roadmap.md` 和 `docs/repository-layout.md`，将下一步调整为 recovery wrapped material 读取接口、metadata API 和错误语义验证。
- 已补 `LatestRecoveryWrappedMaterial` storage 接口，内存 store 与 SQLite store 可返回当前 active recovery metadata 和 encrypted wrapped material bytes，并复验 `wrapped_material_len` / `ciphertext_hash`。
- 已补 storage conformance happy path 和 SQLite 缺失 recovery wrapped material blob 的 `storage_unavailable` 测试。
- 已同步 `docs/sync-server-api-storage.md`、`docs/technical-plan.md`、`docs/roadmap.md` 和 `docs/repository-layout.md`，将下一步调整为 metadata API、统一错误响应、限速和日志脱敏验证。

## 风险与未完成项

- 当前仍不实现 HTTP handler、Docker Compose、真实远端上传下载、Rust 远端 sync client、Flutter manager 或平台壳。
- Device wrapping record 已保存 wrapped key metadata 与 encrypted wrapped key bytes；真实授权 handler 对外开放时仍必须只返回密文 bytes 和服务端可见 metadata。
- Recovery latest 已能返回 metadata 与 encrypted wrapped material bytes；真实恢复 handler 前仍需补齐限速和日志脱敏测试。
- Go recovery record 现在持久化 KDF 参数以支持验签；后续 HTTP DTO 和 Rust 远端客户端接线时必须继续保持这些字段只描述加密恢复材料，不包含恢复码明文或 KDF 输出。

## 下周建议

- 同步补统一错误响应、恢复读取限速和日志脱敏测试。
